The Two Paths

When starting a React Native project, you have two main approaches: Expo and Bare React Native. Think of it like buying a car ‚Äî you can get a fully-loaded model with all the features included, or you can get a base model and add what you need yourself.

Both paths lead to production apps ‚Äî the difference is in the development experience

What is Bare React Native?

Bare React Native (sometimes called "vanilla" React Native) is the framework as Meta provides it. You get:

  • The core React Native framework
  • Full access to the ios/ and android/ native folders
  • Complete control over native configuration
  • The need to manually integrate every native feature

It's like getting a house with just the foundation and framing ‚Äî you add the plumbing, electrical, and HVAC yourself. Maximum flexibility, maximum work.

What is Expo?

Expo is a platform built on top of React Native. It provides:

  • A pre-configured development environment
  • The Expo SDK ‚Äî dozens of pre-built native modules
  • Expo Go ‚Äî an app for testing without native builds
  • EAS (Expo Application Services) ‚Äî cloud builds, updates, and submission
  • Expo Router ‚Äî file-based navigation

It's like getting a fully-furnished house. You can still renovate and customize, but you can also move in and start living immediately.

üì¢ The Official Recommendation

As of 2024, the React Native documentation officially recommends Expo as the way to start new projects. This is a significant endorsement ‚Äî Meta is saying "use Expo" in their own docs.

The Evolution of Expo

If you've heard negative things about Expo, they're probably outdated. Expo has evolved dramatically, and the platform of 2024 is fundamentally different from the Expo of 2018 or 2020.

The Old Complaints (Now Resolved)

"You'll have to eject eventually" ‚úÖ Ejecting is almost never needed now
"Expo apps are too large" ‚úÖ Modern Expo apps have comparable sizes

"You're locked into Expo's ecosystem" ‚úÖ You can always add/access native code

‚úÖ The Key Insight

Modern Expo isn't a "walled garden" ‚Äî it's a set of tools that work together. You get the convenience of pre-built modules AND the ability to add custom native code when needed. It's additive, not restrictive.

What Changed?

The biggest shift was the introduction of Development Builds and the maturation of EAS Build. These two features eliminated the main pain points:

üî® EAS Build

Build iOS and Android apps in the cloud. No Mac required for iOS. Handles code signing. Makes building as easy as eas build.

üì± Development Builds

Create a custom version of "Expo Go" that includes any native modules you need. Get the convenience of Expo Go with unlimited native access.

Development Builds: The Best of Both

Development Builds are the secret sauce that made Expo viable for production apps. They deserve special attention because they eliminate the historical trade-off between convenience and capability.

What is a Development Build?

Think of Expo Go as a "generic" app that can run any Expo project ‚Äî but it can only include the native modules that Expo decided to bundle. A Development Build is a custom app built specifically for your project, including whatever native modules you need.

How It Works

  1. Configure your project ‚Äî Add the native modules you need to your project
  2. Build once ‚Äî Run eas build --profile development in the cloud
  3. Install on device ‚Äî Install the resulting app on your phone/simulator
  4. Develop normally ‚Äî The dev build connects to your Metro bundler, just like Expo Go
  5. Rebuild when native code changes ‚Äî Only needed when adding new native modules

‚úÖ The Magic

You only rebuild when your native code changes. JavaScript changes still hot-reload instantly. So 95% of your development feels exactly like using Expo Go ‚Äî fast and seamless.

Making Your Decision

So when should you use Expo vs Bare React Native? Let's make this concrete:

Use Expo (Recommended) When:

  • ‚úÖ Starting a new project (almost always)
  • ‚úÖ You want fast iteration and good developer experience
  • ‚úÖ You don't have a Mac but need iOS builds
  • ‚úÖ You want over-the-air updates
  • ‚úÖ You want simplified app store submission
  • ‚úÖ Your team includes less experienced native developers
  • ‚úÖ You're building a typical app (content, social, e-commerce, productivity)
  • ‚úÖ You need the Expo SDK modules (camera, location, etc.)

Consider Bare React Native When:

  • ‚ö†Ô∏è You're integrating with an existing native app ("brownfield")
  • ‚ö†Ô∏è You have specific native build requirements that EAS can't meet
  • ‚ö†Ô∏è Your organization requires on-premise builds (security/compliance)
  • ‚ö†Ô∏è You have an experienced native team who wants full control
  • ‚ö†Ô∏è You're deeply customizing the React Native framework itself

Can You Switch Later?

üí° Yes, You Can!

Starting with Expo doesn't lock you in. You can:

  • Add custom native code via Development Builds at any time
  • "Prebuild" to generate native folders if you need direct access
  • Continue using Expo tools even with custom native code

The modern Expo workflow is designed to grow with your project, not constrain it.

Summary

üéâ Key Takeaways

  • Expo is now officially recommended ‚Äî The React Native docs endorse it as the starting point
  • Old complaints are outdated ‚Äî Development Builds solved the "can't use custom native code" problem
  • Expo Go is for prototyping ‚Äî Development Builds are for production apps with custom needs
  • EAS provides cloud services ‚Äî Builds, updates, and submission without complex CI setup
  • Bare RN is for special cases ‚Äî Brownfield apps, on-premise requirements, or deep native customization
  • You're not locked in ‚Äî Expo projects can add native code at any time

üìö Our Course Approach

Throughout this course, we'll use Expo with the following strategy:

  1. Start with Expo Go ‚Äî For quick setup and learning
  2. Move to Development Builds ‚Äî When we add features requiring native code
  3. Use EAS Build ‚Äî For production builds and deployment

This gives you the best of both worlds: fast iteration during learning, and production-ready skills for real apps.
