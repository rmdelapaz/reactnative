The Ecosystem Layers

The React Native ecosystem can feel overwhelming at first ‚Äî there are so many packages, tools, and options. Let's break it down into layers so you can understand where everything fits.

Think of it like a city. At the foundation, you have the infrastructure (roads, utilities). Then you have different neighborhoods (frameworks), each with their own shops and services (libraries). Understanding the layout helps you navigate efficiently.

The React Native ecosystem as a layered stack

Layer by Layer

üèóÔ∏è Native Platforms (Foundation)

At the bottom, we have iOS and Android ‚Äî the actual operating systems your app runs on. You rarely interact with this layer directly when using Expo, but it's good to know it's there.

‚öõÔ∏è React Native Core

Maintained by Meta, this is the framework itself. It provides the fundamental components (View, Text, Image) and APIs (StyleSheet, Animated). This is what makes React Native... React Native.

üöÄ Expo SDK & Services

A layer of convenience and power on top of React Native. Expo provides pre-built native modules (camera, location, notifications) and cloud services (builds, updates, push notifications). This is where we'll spend most of our time in this course.

üìö UI & Feature Libraries

The broader community ecosystem. These are third-party packages for navigation, state management, UI components, and more. Some are essential (like navigation); others are optional conveniences.

üì± Your Application

Everything comes together here ‚Äî your components, your business logic, your styles. This is what you're building.

Expo: Your Development Superpower

If React Native is the engine, Expo is the fully-loaded vehicle that comes with GPS, air conditioning, and roadside assistance included. You could drive with just the engine, but why would you?

üìñ What is Expo?

Expo is a platform built around React Native that helps you develop, build, deploy, and iterate on iOS, Android, and web apps from the same JavaScript/TypeScript codebase. It's maintained by Expo (the company), and it's now the officially recommended way to start React Native projects ‚Äî even in the React Native docs.

Expo Go vs Development Builds

Expo offers two ways to run your app during development:

Feature Expo Go Development Build
Setup Download app, scan QR Build custom app once
Speed to start ‚ö° Instant üïê Initial build takes time
Native modules Only Expo SDK Any native code

Custom native code ‚ùå Not supported ‚úÖ Full support

‚úÖ Our Approach in This Course

We'll start with Expo Go for rapid learning and prototyping. Later, when we need custom native modules or prepare for production, we'll switch to Development Builds. This is the recommended path ‚Äî it lets you move fast early and scale up when needed.

EAS: Expo Application Services

EAS is Expo's cloud service suite. The three main services are:

üî® EAS Build

Build your app in the cloud. Get iOS builds without owning a Mac. Handles code signing automatically.

üì§ EAS Submit

Submit to App Store and Google Play directly from the command line.

üîÑ EAS Update

Push JavaScript updates instantly without app store review (over-the-air updates).

The "Ejecting" Question

You might have heard old advice about "ejecting" from Expo. This is outdated. The modern Expo workflow rarely requires ejecting. Here's why:

‚ö†Ô∏è Myths vs Reality

  • Myth: "Expo is limiting" ‚Üí Reality: Expo supports custom native code via Development Builds
  • Myth: "You'll have to eject eventually" ‚Üí Reality: Most apps never need to eject
  • Myth: "Expo apps are bigger" ‚Üí Reality: Modern Expo apps are comparable in size
  • Myth: "Expo is just for beginners" ‚Üí Reality: Major companies use Expo in production

Library Evaluation Checklist

When evaluating a third-party library, ask:

‚úÖ Before Adding a Library

  1. Is it maintained? ‚Äî Check last commit date, open issues, release frequency
  2. Is it popular? ‚Äî npm downloads, GitHub stars (not everything, but a signal)
  3. Does it work with Expo? ‚Äî Check docs or React Native Directory
  4. What's the bundle size impact? ‚Äî Use bundlephobia.com to check
  5. Is the API well-documented? ‚Äî Poor docs = pain later
  6. Does it support the New Architecture? ‚Äî Increasingly important
  7. Are there alternatives? ‚Äî Sometimes built-in or simpler solutions exist

The "Minimal Dependencies" Philosophy

‚ö†Ô∏è Resist the Urge to Over-Install

Every dependency is a liability ‚Äî it can break, become unmaintained, or cause version conflicts. Before adding a library, ask: "Can I reasonably build this myself in under an hour?" If yes, consider doing so. Your future self will thank you.

Summary

  • Be selective with dependencies ‚Äî Prefer built-in solutions, evaluate libraries carefully
